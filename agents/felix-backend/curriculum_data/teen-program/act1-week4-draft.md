## Section 2 — What the Student Brings In

The student enters Week 4 with exactly one thing: their identity card from Week 2 — unchanged since they wrote it, tested under pressure in Week 3, and now being used on familiar ground.

**What the identity card contains:**
- Declaration 1 (Persona): who the agent is, what voice it has, what audience it's built for
- Declaration 2 (Scope): what the agent refuses
- Declaration 3 (Behavior instructions): three clarifying questions the agent asks before answering, in numbered order

**What the student experienced in Week 3:**
- Their identity card HELD on voice and consistency (for Type A students)
- Their identity card DID NOT hold on series structure — each of the three posts started fresh, with no connection to the others
- They heard the named gap: "Your agent has an identity. What it doesn't have yet is a workflow — instructions for how to work through a task step by step."
- They heard the seed: "Next session, we're going to give it instructions for HOW to work — step by step. That's what makes a series feel like a series."

**Why the Week 3 workflow gap doesn't matter this week:**

Week 3's harder task was a three-post series. That task exposed the Workflow gap — the identity card can't build a series because it has no instructions for how posts should connect to each other.

Week 4's task is the Week 1 task: a single output — one niche recommendation. The Workflow gap is irrelevant on a single-output task. Students win not because the task is easy, but because their Week 2 fix was the right fix for the Week 1 failure.

**Two card types entering Week 4:**

*Type A (strong card, clear win expected):*
- Specific stance, clear audience, priority trade-off in Persona
- Concrete refusals in Scope
- Literal, numbered clarifying questions in Behavior
- Week 3 Part A score held at 40–70
- Expected in Week 4: consistent recommendation across both runs, same questions asked, single clear recommendation
- Expected win: beats Screen 2 on 4–5 of 5 dimensions

*Type B (weak card, partial win expected):*
- Adjective-inflated Persona ("helpful, creative, knowledgeable")
- Vague Scope
- May still have literal Behavior questions (the one deterministic element — even a weak card can have these)
- Week 3 Part A score remained low (10–35)
- Expected in Week 4: better than Week 1's Screen 2 on at least Questioning behavior; still inconsistent on voice and specificity
- Expected win: beats Screen 2 on 2–3 of 5 dimensions; the win is partial but real

**Facilitator note before Week 4 opens:** Check your Week 3 facilitator capture form. Know which students are Type A and which are Type B before the session opens. Fastest signal: Week 3 Part A Consistency score below 10 = Type B. Path determination affects Beat 2 and Beat 3 scripts. If you didn't score Week 3 before this session, check the Week 3 facilitator capture form's (a) field — it will tell you which students were placed on Path B.

## Section 4 — Four-Run Protocol

**What the four runs are:**

| Run | Agent | Operator |
|---|---|---|
| Run 1 | Student's identity-card agent | Student (Beat 1 Step 1) |
| Run 2 | Student's identity-card agent | Student (Beat 1 Step 2) |
| Run 3 | Screen 2 (generic AI, no system prompt) | Platform-automated |
| Run 4 | Screen 2 (generic AI, no system prompt) | Platform-automated |

**Why four runs, and what each reveals:**

**Runs 1 + 2 — student's agent:**
The identity card is the only active variable. Running twice confirms whether the card holds — same recommendation both times, or drift? For Type A students: both runs should return the same core recommendation (or very close) with the same clarifying questions asked each time. For Type B students: some variance may appear, but Questioning behavior (the one deterministic element of any card with literal questions) should hold across runs regardless.

**Runs 3 + 4 — Screen 2:**
Screen 2 is the baseline the student started with. Running it twice on the same task produces what it has always produced: different recommendations, no clarifying questions, invented statistics. Screen 2 hasn't changed. Two runs make this visible the same way two runs made the Week 1 failure visible — and that's the point.

**Why Screen 2 runs are platform-automated in Week 4:**
Students spent Week 1 operating Screen 2 to experience the failure firsthand. Week 4 is WIN week — the student's experience belongs entirely in their own agent. Having students manually operate Screen 2 again would re-create the Week 1 register, which is the wrong emotional frame for a WIN session. The platform runs Screen 2 in the background so the comparison is ready when Beat 2 opens, without splitting the student's attention or pulling them back into the failure experience.

If the platform cannot automate Screen 2 runs: the facilitator should run Screen 2 manually on the same prompt before the session opens and have both outputs ready before Beat 2 begins. Do NOT run Screen 2 manually during Beat 1 while students are running their agents — it pulls the facilitator's attention away from observation.

**What to watch for during Beat 1 (facilitator only — do not share with students during the run):**

| What to observe | What it means |
|---|---|
| Agent asks the same clarifying questions in both runs | Behavior instructions holding — deterministic layer working |
| Agent gives same core niche recommendation in both runs | Persona and Scope holding — strong card (Type A) |
| Agent gives different recommendation between runs | Weak Persona — likely Type B; adjective inflation producing variance |
| Agent gives one recommendation, not a list | Scope or Persona includes a "one recommendation only" constraint |
| Student looks satisfied after completing both runs | Natural response for Type A students — let it happen |
| Student looks uncertain or disappointed | Worth noting; may be Type B or unexpected variance; Beat 2 handles it |

Do not comment on observations during Beat 1. Students run; facilitator records.

**Platform requirement:** Save all four run outputs per student before Beat 2 opens. Beat 2 surfaces them in a structured panel alongside Week 1 stored outputs. The evaluator agent also needs all four Week 4 outputs to generate the leaderboard scores for Beat 3.

## Section 5 — Beat 2 — Three-Layer Reveal + Comparison Script

**Trigger:** All students have submitted their two agent runs.

**Platform action:** Beat 2 opens. The platform surfaces all relevant outputs in a split panel. Students can see:
- Their Week 1 Screen 2 outputs (stored from Week 1, two runs — retrieved by task ID)
- Their Week 4 student agent outputs (just submitted, two runs)
- Screen 2's Week 4 outputs (platform-automated, two runs)

**The three-layer structure — what the facilitator reveals, in this exact order:**

> **Layer A: Screen 2 now vs Screen 2 in Week 1** — establish that the benchmark hasn't changed
> **Layer B: Student agent now vs student in Week 1** — establish that the student has changed
> **Layer C: Student agent now vs Screen 2 now** — the WIN comparison

**Why this order matters (facilitator context only — not on screen):**

Layer A first: before the win lands, students need to know the benchmark is still the same thing they faced in Week 1. If Screen 2 had somehow improved, the comparison would be ambiguous. Layer A establishes that what they're beating is exactly the same target.

Layer B before Layer C: the win is more meaningful if students first see that THEY changed, not only that they beat a fixed target. Layer B is about personal growth — "look at what you produced in Week 1 versus now." Layer C is about the competition — "you beat Screen 2." Seeing Layer B first gives Layer C emotional weight. Without Layer B, the win can feel like defeating an easy target. With Layer B, the win feels like becoming something different.

Do not rush to Layer C. Let Layer B land first.

## Section 10 — Scoring Instrument — Week 4 Final Version

**Purpose:** Score each student's identity-card agent and Screen 2 on the same five dimensions used in Weeks 1–3. This generates the leaderboard scores, the growth delta, and the cohort-level Act 1 record. Consistency of the instrument across all four weeks is what makes the delta meaningful.

**When to score:** The evaluator agent generates scores after all four runs are submitted and before Beat 3 opens. Scores must be ready before the leaderboard ceremony begins. The facilitator reveals scores — they do not generate them live.

**Platform requirement:** The evaluator agent receives the four run outputs per student and returns a structured score object: `{student_agent: {consistency, specificity, local_grounding, questioning, claim_integrity, total}, screen2: {same_fields}}`. The platform also retrieves each student's Week 1 Screen 2 baseline score (stored from Week 1) for delta calculation.

---

### Scoring Instrument — Niche Recommendation Quality (0–20 per dimension, 0–100 total)

*Same instrument as Weeks 1–3 for direct cross-week delta comparison. Apply unchanged.*

| Dimension | 0 | 10 | 20 |
|---|---|---|---|
| **Consistency** | Runs 1 and 2 gave a different top niche recommendation | Runs overlapped in some categories but different lead recommendation | Runs gave the same core recommendation |
| **Specificity** | Generic list, no single recommendation | Single recommendation with vague or generic reasoning | Single recommendation with specific, concrete reasoning |
| **Local grounding** | No meaningful MY/SG reference | MY/SG mentioned but advice is generic | Specific, checkable local trend data used to support recommendation |
| **Questioning behavior** | No clarifying questions before answering | One clarifying question asked | Two or more relevant questions asked before answering |
| **Claim integrity** | Multiple specific statistics cited as fact (unverifiable) | Mix — some claims with source, some invented | All claims checkable, or AI stated uncertainty appropriately |

---

### Score Protocol

1. Score BOTH student agent runs (Runs 1 and 2) on all five dimensions.
2. Score BOTH Screen 2 runs (Runs 3 and 4) on all five dimensions.
3. For each agent (student and Screen 2): use the **lower** of the two run scores per dimension as the official score. The lower score captures the real consistency level, not the lucky-run level.
4. Sum five dimension scores for the official total (0–100).
5. Retrieve the student's Week 1 Screen 2 baseline score (stored from Week 1 scoring protocol).
6. Calculate growth delta: Week 4 agent total − Week 1 baseline total.
7. Calculate leaderboard score: (Week 4 agent total × 0.60) + (growth delta × 0.40).

---

### Evaluator Agent Scoring Rubric

*Instructions for the evaluator agent — to be included in the evaluator system prompt. Apply these criteria consistently across all four runs before returning scores.*

**Consistency:** Compare the lead niche recommendation across the two runs for the agent being scored. If the top recommended niche (or clearly primary recommendation) is the same in both runs — 20. If the runs share some niche categories but differ on the specific lead recommendation — 10. If the top recommended niche differs between runs — 0.

**Specificity:** Look at the final recommendation in the lower-scoring run. If it is a single recommendation with a concrete, specific reason grounded in something the student or agent stated (for example: "focus on travel vlogs because you mentioned you travel frequently and MY/SG travel content on TikTok tends to get higher save rates than generic lifestyle content") — 20. If it is a single recommendation with a vague or generic reason ("focus on lifestyle content because it's popular") — 10. If it is a list of options without a primary recommendation — 0.

**Local grounding:** Look at how MY/SG is referenced. If the output cites specific, checkable local information — named places, named platforms, specific behaviors or patterns specific to MY/SG, not just saying "in Malaysia and Singapore" — 20. If MY/SG is mentioned but the advice is applicable anywhere ("social media content is growing fast in Malaysia") — 10. If there is no meaningful MY/SG reference — 0.

**Questioning behavior:** Count the clarifying questions asked before the agent gave its recommendation. Two or more questions — 20. One question — 10. Zero questions — 0.

**Claim integrity:** Look at all statistics, percentages, and trend claims in the output. If all claims are either qualified ("based on general trends"), omitted entirely, or verifiably grounded — 20. If the output has a mix of qualified and unqualified statistics — 10. If the output includes multiple specific statistics stated as fact (e.g., "beauty content grew 47% year-on-year in Malaysia") without source attribution — 0.

**Calibration protocol:** Score each run individually on all five dimensions first. Then apply the lower-score protocol — for each dimension, record the lower of the two run scores as the official score. Return: `{run1: {all_dimensions}, run2: {all_dimensions}, official: {lower_per_dimension}, total: sum_of_official}` for each agent.
